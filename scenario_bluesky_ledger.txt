A 200-employee B2B SaaS company called BlueSky Ledger runs entirely on Azure and Microsoft 365. Its core product is a multi-tenant web application that provides financial workflow automation to mid-market customers. The web app is hosted on Azure App Service, with a PostgreSQL database on Azure Database for PostgreSQL. Authentication for the app uses Azure AD / Entra ID, with SSO enabled for customer tenants as well as internal staff.

The workforce is fully remote across the US and EU. All employees use company-managed laptops with Windows 11 and Microsoft Defender, but local administrator rights are granted to engineers by default for developer tooling. Contractors are used heavily for support and implementation work, and contractor accounts are provisioned manually via Azure AD with no standardized offboarding checklist.

The company uses Microsoft 365 (Exchange Online, SharePoint, OneDrive, Teams) and relies on email and Teams for most customer communication. There is no dedicated security team; security responsibilities are split between a small IT team (3 people) and the Head of Engineering, who handles security architecture part-time.

Production deployments are done via GitHub Actions into Azure. Secrets for CI/CD and app configuration are stored partly in Azure Key Vault and partly as GitHub Actions secrets, but there is no formal secrets inventory or rotation policy. Some legacy secrets still exist in appsettings.json files checked into private repos. There is no centralized SIEM; the team occasionally reviews Azure Activity Logs and M365 alerts in their respective portals, but there is no unified log aggregation or alert triage process.

The public-facing web app exposes a small number of REST APIs and a browser-based UI. API authentication uses JWT access tokens issued by Azure AD, but session management and token revocation logic are homegrown and have not been formally threat-modeled. The app includes a "global admin" role for internal staff that can access any customer tenant's data for support and troubleshooting. Admin actions are logged in the application database, but there is no routine review of these logs.

Customer data (including invoices, payment instructions, and limited banking details such as IBANs and routing numbers) is stored in the shared PostgreSQL database using logical tenant isolation (tenant_id columns) only—Row-Level Security is not currently enabled. Backups of the database are taken daily using Azure's built-in backup features and stored in Azure Blob Storage in the same subscription. Backup configuration was set up by an engineer several years ago and has not been formally reviewed since.

BlueSky Ledger is in the middle of SOC 2 Type 1 preparation with an external auditor; they have scoped Security, Availability, and Confidentiality as in-scope Trust Services Criteria, but they have not yet started the Type 2 observation period. There is no ISO 27001 certification in progress. A basic security policies deck exists in Confluence, but it is incomplete and has not been rolled out formally to all staff.

No AI or machine-learning features are currently deployed in production; the company uses a hosted LLM tool internally for documentation drafting, but this tool does not have access to production customer data.
